Перелік основних команд Git

  • git config --global user.name blabla - дозволяє вказати конфігураційні налаштування Git (ваше ім’я)
  • git config --global user.email blabal@bla - дозволяє вказати конфігураційні налаштування Git (вашу електронну адресу)
  • git init - створить новий підкаталог .git, який містить всі необхідні файли вашого репозиторія — скелет Git-репозиторія
  • git status - показує стан файлів у робочому каталозі та індексі: які файли змінені, але не додані в індекс; які очікують комміта в індексах
  • git add - додає вміст робочого каталогу в індекс (проміжна область) для наступного комміту
  • git commit - запис змін до сховища, містить поточний вміст індексу та дане повідомлення журналу з описом змін
  • git log - показує історію комітів, починаючи з найсвіжішого та йдучи до витоків проекту. За замовчуванням, вона показує лише історію поточної гілки, але може бути налаштована висновок історії інших, навіть кількох одночасно, гілок. Також її можна використовувати для перегляду різниці між гілками на рівні коммітів
  • git checkout - перемикання гілок або оновленняє файлів в робочому дереві, щоб вони відповідали версії в індексі або вказаному дереві
  • git branch - це свого роду "менеджер гілок". Вона вміє перераховувати ваші гілки, створювати нові, видаляти та перейменовувати їх
  • git push -використовується для вивантаження вмісту локального репозиторію у віддалений репозиторій. Оновлює віддалені посилання, використовуючи локальні посилання, одночасно надсилаючи об’єкти, необхідні для завершення заданих посилань
  • git pull - Git спочатку забирає зміни із зазначеного видаленого репозиторію, а потім намагається злити їх із поточною віткою
  • git merge - використовується для злиття однієї або декількох гілок на поточну. Потім вона встановлює покажчик поточної гілки на результуючий коміт
  • git clone - дозволяє отримати копію існуючого Git репозиторія  у щойно створений каталог, створює гілки віддаленого відстеження для кожної гілки в клонованому сховищі, а також створює та перевіряє початкову гілку, яка розгалужується від поточної активної гілки клонованого репозиторію

.gitignore файли це ігноровані файли, які реєструються у кореневому каталозі репозиторію. У Git немає спеціальної команди для вказівки файлів, що ігноруються: замість цього необхідно вручну відредагувати файл .gitignore, щоб вказати в ньому нові файли, які повинні бути проігноровані. Файли .gitignore містять шаблони, які зіставляються з іменами файлів у репозиторії, щоб визначити необхідність ігнорувати ці файли.

Advanced

  • git revert - скасовує дії минулих коммітів, створюючи новий, що містить всі скасовані зміни. Цей спосіб не змінює історію коммітів. Його зазвичай використовують у публічних гілках, де зміна історії небажано
  • git reset - видаляє всі коміти вище вказаного. Не додає нову інформацію в історію, але теж її змінює
  • git rebase - відбуваєьтся заміна одного комміту в основі гілки на інший, внаслідок чого створюється враження, що гілка отримала новий початок, але насправді гілка завжди складається із абсолютно нових коммітів
  • git cherry - pick - вносить виправлення одним коммітом, і намагається повторно застосувати їх у вигляді нового комміту в поточній гілці. Ця можливість корисна в ситуації, коли потрібно забрати парочку коммітів з іншої гілки, а не зливати гілку цілком з усіма змінами, що внесені до неї
  • git stash - дозволяє «ховати» зміни у робочій директорії та відновлювати їх за необхідності. Ця команда не видаляє файли, вони потрапляють у спеціальне місце всередині директорії .git "на збереження» і не торкається нових файлів, оскільки вони ще не є частиною репозиторію
